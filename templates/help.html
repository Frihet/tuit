Help

This is the main help page for {{ application|escape}}. {{ application|escape }} is The Ultimate ITIL Toolbox, a set of software components designed to support ITIL based work flow for service management. {{ application|escape }} currently consists of the following components:

  • An issue handling system, based on Roundup
  • A CMDB based on FreeCMDB. For more information about the CMDB concept, see this Wikipedia article on CMDBs.
  • A knowledge base, based on Foswiki. For more information about the CMDB concept, see this Wikipedia article on knowledge bases.

User interface overview

This section described the basic user interface concepts of {{ application|escape }}. These concepts are the same in all components, e.g. the Knowledge base looks and works as much like the CMDB as is possible without compromising the functionality of either.

The top menu

The top menu allows you to navigate to the different parts of the site. This menu does not usually change depending on what part of {{ application|escape }} you are using, but because different users have access to different parts of the site, it will vary depending on who is logged in. Top menu is also highly customisable, and information on chnaging top menu components can be found here:

    Changing top menu components

The top menu also allows you to quickly search the system. Just enter a few keywords into the search box and press the «enter» key. For more information on searching, see the search section.

The left side menu

The menu on the left side of every page contains page specific links and options. This menu will be different in e.g. the Knowledge base than in the CMDB.

The main content area

The remainder of the screen is called the main content area, this is where the main application data is shown. What the content area contains depends heavily on what page you are viewing.

The home screen

This is the start page of {{ application|escape }}. A future version of {{ application|escape }} will contain a user configurable dashboard showing various status information and messages, but the current home screen is very limited. It contains the following items:

  • A list of all Incidents assigned to the logged in user, sorted by priority.
  • A list of all Problems assigned to the logged in user, sorted by priority.
  • A list of all RfCs assigned to the logged in user, sorted by priority.
More detailed information on creating and modifying dashboard widgets can be found here. Archive with documentation in doc and html file formats, together with example widgets is also available for download.

Searching {{ application|escape }}

{{ application|escape }} contains a global search function that will search every component of {{ application|escape }}. To quickly perform a search, enter a search word in the search box in the top right corner of any {{ application|escape }} page, and then press the «enter» key.

If a search yields many results, only the first ten results are shown, and the user may press a button in order to see more results.

Search types

{{ application|escape }} shows the following types of search results:

  • IDs. If issue id or multiple ids separated by space are entered in search box, this search type will show them regardles of their status or type.
  • Issues. This shows both Incidents, Problems and RfCs. Freetext searches the subject, the description, all updates as well as the name of the user who requested the issue. All advanced search options work.
  • CIs. This shows all matching items from the CMDB. Freetext searches the CI name. No advanced search options work.
  • Knowledge base articles. This shows all matching items from the Knowledge base. Freetext searches title, the KB form and the KB text proper. No advanced search options work.
  • User search. This shows all matching users. Freetext searches the username, first and last name. No advanced search options work.

Advanced search

To specify additional search options, click on the «Advanced search options» button in a search result page. There, issue priority, update date and other search criteria can be applied. Note that not all types of search filters apply to all search places, e.g. User search ignores all filters except the search text.

{{ application|escape }} Issue Management

The {{ application|escape }} issue handling component is an issue tracker loosely based on Roundup. It is intended to be used to track Incidents, Problems and Requests for Change as they make their way through your IT organization.

Types of issues

{{ application|escape }} is based around 3 types of issues:

  • Incidents are issues that have caused or may in the future cause service interruptions, that need to be handled before breaching an SLA.
  • Problems are issues that support root cause analysis of one or more incidents.
  • RfCs Or Requests for Change, are issues that track roll out of a change in a service, often triggered by a Problem.

Creating a new issue

For each issue type there is a corresponding link in the top menu of {{ application|escape }} for filing an issue of that type. Use one of these links in order to file a new issue. You will be presented with a form with various fields to fill out.

When done, click the «Create issue» button at the bottom of the page, and you are done.

Filling out user fields

Issues have many users associated with them, including the raiser of the issue, the person assigned to a specific issue, etc.. Fields for filling out users in {{ application|escape }} all use autocompletion. The autocompletion code will search the username, first name and last name of the user when performing the completion.

There are several variations on how user fields work:

  • Some user fields only permit a single user, these text field for these fields is only one line long. Others permit arbitrarily many users. These text fields contain multiple lines.
  • Some user fields only allow registered users to be entered, others also allow you to enter an email address. If you've added an email address once, it will be added into {{ application|escape }} as a contact. When receiving an email about an issue, a contact is also created. These contacts are also used for autocompletion where appropriate, and they can also be edited in the administration back end.

Filling out dependencies to other issues

Issues can depend on other issues. The field for changing issue dependencies uses autocompletion. The autocompletion code will search the issue subject, description and the text of all updates.

Filling out dependencies to CIs

Issues can depend on configuration items (CIs), that is, the items stored in the CMDB portion of {{ application|escape }}. The field for changing CI dependencies uses autocompletion. The autocompletion code will search the name of the CI. There is, however, also a second method for specifying a CI, the CMDB drill down feature. This is a tree of all known CIs, that can be selectively expanded to drill down through dependencies until the wanted one is found. See the section on CMDB navigation for more information.

Using quick fills to automatically fill out common fields

On many service desks, there is a great number of repetitive tickets. This can be caused by a common underlying problem that needs to be fixed or simply because users tend to run into the same types of problems. In order to simplify the creation of such repetitive issues, {{ application|escape }} provides a feature called Quick fills. This is simply a drop down box with common issue «templates». Select one of them, and various fields will be automatically filled out for you. Exactly what fields are set to what values on every Quick fill can be changed in the administration back end.

See the section Changing the available quick fills for information on customizing the list of available quickfills.

Using generic fills to fill out fields depending on given condition

In many cases values of some fields are always the same depending on certain condition. For example if issue is registered by certain user, urgency of the issue is always one determined value. {{ application|escape }} provides a feature called Generic fills in order to automate this process. This is just a simple set of rules, that define field values depending on certain conditions.

See the section Changing available generic fills for information on customizing the list of available generic fills.

Creating a new issue based on another issue

It is also possible to create a new issue by basing it on another issue. Perform the following steps to create a new issue this way:

  1. First go the the web page of the original issue. A link to the relevant page should be included in every email from the {{ application|escape }} system, or you could find the update by searching {{ application|escape }} or browsing your home page.
  2. Once you are on the correct page, click the «Create new ticket based on this ticket» button at the top of the screen.
  3. This will show a tiny dialog on the screen where you will be asked what type of issue to create. Pick the desired issue type from the drop down box and press the «Create» button.
  4. The same issue creation form that is displayed when clicking one of the issue creation links in the top menu will now be shown, but with the following fields copied from the original issue:
    • Issue subject
    • Description of the issue
    • Issue category
    • Impact of issue
    • Urgency of issue
    • Requester of issue
    • User assigned to this issue
    • People CC:ed on this issue
    • Users who are co-responsible for this issue

Updating an issue

There are two basic ways to update in issue. The first is through the web interface, the other is through email.

Updating an issue through email

In order to create an update through email, simply reply to an email sent from {{ application|escape }}. The content of the email, including any attached files, will be added as an update to the issue. If the issue was closed, it will be reopened, but otherwise, no field values are changed.

Updating an issue through the web

Updating an issue through the web interface allows more fine grained control over an update. It is possible to change the values of all fields of an issue except the subject and the original problem description. It is possible to explicitly specify who should receive email notification of the update, and it is possible to make the update internal, e.g. hide it from non-privileged users.

To update an issue, first go the the web page of that issue. A link to the relevant page should be included in every email from the {{ application|escape }} system, or you could find the update by searching {{ application|escape }} or browsing your home page. Once you are on the correct page, click the «Add new update» button at the bottom of the screen. For more information about changing specific types of fields, such as auto completing user name fields, see the section on creating issues.

Using an issue update as the basis for a knowledge base article

When having solved an issue, it is sometimes desirable to make that solution available to others as a knowledge base article. This has the potential to greatly reduce the amount of time required to solve similar issues in the future. For this reason, {{ application|escape }} makes it as easy as possible to perform this task. The following steps are all that is required:

  1. Locate the issue containing the relevant update and scroll down to find the correct update.
  2. Click on the «Create KB article from this update» button.
  3. A small dialog is presented, allowing you to choose the name of the article (The default is the subject of the ticket) and to specify which section of the knowledge base to use. Press the «Submit» button once you are done.
  4. Fill out the Knowledge base article form and make any changes to the main article text that are needed. Press the «Save» button when you are done to create the Knowledge base article.

Using a knowledge base article as the basis for an issue update

A knowledge base is only useful if it is used. For this reason, {{ application|escape }} makes it as easy as possible to use Knowledge base information when updating an issue. Below the text field for making an update comment, there is a text input field. When writing into this field, {{ application|escape }} will search the Knowledge base, and show you an autocompletion list of articles whose subject or content match that text. Once an article has been chosen, the user can either press the «Preview» button in order to see a preview of the article contents or press the «Insert» button in order to insert the article contents into the update.

{{ application|escape }} CMDB

Navigating the CMDB

Usually, the easiest way to navigate the CMDB is through dependencies. The CMDB start page contains a list of suitable start nodes. Each of these nodes has a small button next to them with a + sign. Click on the plus to expand and show all nodes related to that node. Using this method repeatedly, it is possible to drill down until the desired CI is located. Then, simply click on the CI to access the main page of that CI.

When like adding a new CI dependency, either in the CMDB or in the issue tracker, the same drilldown interface is also available.

Changing the starting points of navigation in the CMDB

In order to change the list of CIs displayed at the start of the drill down tree, simply update the dependencies of the CI called «Root».

Creating and updating CIs

Creating CIs is easy - simply press the «Create new item» button in the left side menu, and start editing the item. Updating an existing item can be done in two ways:

  • Press the «Update» button of an individual field on the CI to edit that field.
  • Press the «Edit item» link in the left side menu to obtain a form where all fields of the CI can be edited at once.

Using the history feature

Every change ever performed to a CI is saved. To view the performed changes, follow the «History» link in the left side menu of a CI. A list of all changes will be displayed. To view a specific revision of a CI, you can click the ½Show» button next to that revision. It is also possible to revert all edits performed after a specific edit.

Configuring CI types

The types of CIs (e.g. Server, Service, Firewall) that are available can be configured freely. Perform the following steps:

  1. Access the administration interface. {% if user.is_staff %} It can be found here. {% endif %}
  2. Click the CMDB Administration link.
  3. Click the sidebar item «CI types»
  4. Update the list of CI types as you require, and press «Update».

Configuring CI fields

The fields availalbe to various CI types can be configured freely. Perform the following steps:

  1. Access the administration interface. {% if user.is_staff %} It can be found here. {% endif %}
  2. Click the CMDB Administration link.
  3. Click the sidebar item «CI columns»
  4. Update the list of CI columns as you require, and press «Update».

It is strongly recommended that the default column (which is used for CI description, etc.) be of the unformated text type.

Changing the content of emails sent by {{ application|escape }}

{{ application|escape }} is designed to be usable either through the web interface or through email. When an issue is updated through email, an update is sent to the people assigned to that ticket. In the web interface, the user can specifically choose which parties should recive that particular update. But for {{ application|escape }} to be able to send these emails, one must first specify template emails. This is done through the administration interface.

Take the following steps:

  1. Access the administration interface.
  2. Click on the «change» link next to the item labeled Email templates.
  3. Select which template you wish to change.
  4. Edit the template. The subject is in plaintext, the body is HTML formated. A future version of {{ application|escape }} will include a graphical html editor.
  5. Click the save button.

Autocompletion

Autocompletion or automatic completion, is technique used in many places in {{ application|escape }}. It is useful when picking between a very large selection of items, e.g. in situations where a drop down box is unsuitable, because the number of items in the list would be prohibitive. Autocompletion uses a simple text input field, where the user can enter text, and the entire list of items will be searched, and any matching items will be shown. The user can then use the up and down arrows to move around in the list and press the enter key to select the highlighted item.

CMDB

CMDB: Configuration Managment DataBase, a tool for tracking important assets in an IT environment and the relationships between them. The basic elements of a CMDB are called Configuration Items (CI). A CI can be a physical server, a software service, or even a person. Any item that you want to track using the CMDB is a CI.

Incident

An Incident is an unplanned interruption or a reduction in the quality of an IT Service. A failure that does not impact the Service, such as failure of one disk from a mirror set, is also an Incident and should be reported as such.

Issue

An Issue in {{ application|escape }} represents either an Incident, a Problem or an RfC. Though different types of issues may have different fields and other minor differences, they behave mostly the same. For more specific information about how to create, update and close issues, see the section {{ application|escape }} Issue Management.

ITIL

Information Technology Infrastructure Library or ITIL is a framework for organising IT Service Provider organizations. It is based on best-practices in the IT industry and provides a set of concepts and policies for managing IT infrastructure, development and organisation.

Because {{ application|escape }} is meant to be used to support an ITIL-based workflow, a few of the core concepts of ITIL are described in this document. But this document is about the {{ application|escape }} software, and not about ITIL in general. For a complete overview of ITIL, including information about how to apply ITIL to your organization, please see the IT Frameworks Wiki.

Knowledge base

Knowledge base: A collection of product and/or site specific information, based on experiences by the community members or site personell supplementing official documentation. For more information on ITIL concepts, see the IT Frameworks Wiki.

Problem

A Problem is the underlying cause of one or more Incidents. Problems are to be handled by Problem Management, but if the solutions involve Changes they must be approved by Change Management before being implemented. For more information on ITIL concepts, see the IT Frameworks Wiki.

RfC

A Request for Change or RfC is an official request for a Change to be made. The proposal includes details of the Change, including a detailed roll-back plan, and is sent to Change Management. For more information on ITIL concepts, see the IT Frameworks Wiki.
